# Phase 2 Engineering Report: Consensus ADMM Implementation

## 1. System Architecture
The firmware is built on a **dual-core asynchronous architecture** to meet strict real-time deadlines:
- **Core 0 (Control):** Runs the 100Hz PI control loop, samples the LDR, and executes the ADMM optimization state machine.
- **Core 1 (Communication):** Dedicated entirely to the MCP2515 CAN controller. It handles interrupt-driven packet reception and manages non-blocking transmission queues to prevent network latency from jittering the control loop.

## 2. File-by-File Breakdown

### Firmware (`main/`)
- **`main.ino`**: The system orchestrator. Manages the dual-core boot handshake, the network-wide calibration state machine, and the primary control loop.
- **`admm.cpp / .h`**: The "brain" of the project. Implements the Primal (x), Consensus (z), and Dual (λ) updates. Includes a constrained QP solver that evaluates boundary candidates (C1, C2, C3) to find the cheapest feasible duty cycles.
- **`commands.ino`**: Extends the serial interface with Phase 2 capabilities, including manual ADMM triggers (`T`), identity verification (`i`), and background overrides (`b`).
- **`can_comms.h`**: Defines the 11-bit CAN protocol. Packs 3 floating-point duty cycle proposals into a single 8-byte frame using 16-bit quantization to maximize throughput.
- **`pid.h`**: Implements a robust PI controller with anti-windup. It tracks the reference $r_i$ generated by ADMM.
- **`metrics.h`**: Real-time calculation of Energy ($E$), Visibility Error ($V$), and Flicker ($F$) directly on the silicon.

### Tooling
- **`test_suite.py`**: A comprehensive automation framework. It uses logical identity detection to map physical ports to software nodes and automatically generates research-grade plots.
- **`live_plot.py`**: A multi-process visualization tool. Uses Python's `multiprocessing` to bypass the GIL, ensuring that high-speed serial data doesn't cause GUI lag.

## 3. Consensus ADMM Implementation
We solve the following optimization problem across the network:
$$\min \sum c_i u_i \quad \text{s.t.} \quad K \mathbf{u} + \mathbf{d} \ge \mathbf{L}, \quad 0 \le u_i \le 1$$

### Integration Logic (The "Eq 13" Secret)
To prevent the PI controller from fighting the ADMM during the 12-second convergence window, we implemented **Equation 13**:
$$r_i = d_i + k_{ii} u_i^*$$
By setting the PI reference to only include the local node's intended contribution, the system remains stable even if neighbors are slow to respond on the CAN bus.

## 4. Test Results Analysis

### Occupancy Sweep (Tests 1-5)
- **Test 1 (All Off):** Validates the energy-saving baseline. Duty cycles correctly settled at 0.000.
- **Tests 2-4 (Single High):** Verified the **Identity Mapping**. When Node 1 was requested HIGH, Desk 1 hit 19.99 Lux. This proves the `UID_TABLE` is now correct.
- **Test 5 (All High):** The "Stress Test." All nodes successfully coordinated to hit 20 Lux. 

### Cost Sensitivity (Tests 7-10)
This is where the "Distributed Intelligence" shines.
- **Test 7 (Node 2 Expensive):** Node 2 had a cost of 10. In response, **Node 1 increased its output to 35.3 Lux**. 
- **The Logic:** Node 1 "helped" its neighbor because Node 1's energy was 10x cheaper. By over-illuminating Desk 1, Node 1 provided enough spillover to allow Node 2 to dim down, saving the network money.

### Convergence Research
The `plot_convergence_research.png` shows that the system reaches its optimal state in roughly **5 to 10 iterations**. Even though we allow up to 150, the "heavy lifting" of the consensus is done almost instantly, proving the efficiency of the ADMM approach for real-time lighting.
